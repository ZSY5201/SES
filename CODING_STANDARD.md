# CODING_STANDARD

## 适用范围

本规范适用于 SES 中的 PLC、C#、Python、控制算法、通信程序和辅助工具代码。

## 通用原则

- 命名清晰，不使用含糊缩写。
- 一个函数、方法或功能块只承担一个主要职责。
- 禁止无解释的 Magic Number。
- 所有关键参数必须可配置。
- 所有控制输出必须限幅。
- 所有通信和设备交互必须处理超时。
- 所有故障必须可诊断。
- 默认中文注释，说明工程意图和关键边界。

## 命名规则

命名应表达工程含义，而不是只表达数据类型。

推荐：

- `targetDepth`
- `currentHeading`
- `maxOutputLimit`
- `communicationTimeout`
- `isSensorHealthy`

不推荐：

- `x1`
- `tmp`
- `flag`
- `data`
- `value2`

## 参数管理

参数应集中定义，避免散落在逻辑中。

参数至少应说明：

- 名称
- 默认值
- 单位
- 允许范围
- 工程含义

## 控制输出

所有输出到执行器、阀门、电机、推进器或外部设备的值，都必须考虑：

- 最小值
- 最大值
- 变化率限制
- 故障时默认输出
- 手动模式优先级

## 状态与故障

复杂模块应显式定义状态。

典型状态包括：

- Idle
- Init
- Running
- Hold
- Fault
- Reset

故障信息应能回答：

- 哪个模块故障？
- 什么时候发生？
- 当前输入是什么？
- 当前输出是什么？
- 是否可以自动恢复？
- 是否需要人工确认？

## PLC 规范

- 一个 FB 对应一个明确功能。
- IO 映射与控制逻辑分离。
- 自动逻辑与手动逻辑分离。
- 故障判断与控制输出分离。
- 每个 FB 应包含输入、输出、参数、内部状态和故障信息。
- 关键输出必须有默认安全值。

## C# 规范

- 业务逻辑、通信逻辑和界面逻辑分离。
- 异步通信必须处理取消、超时和异常。
- 日志应包含设备名、操作、结果和异常信息。
- 配置应使用结构化文件或配置对象管理。

## Python 规范

- 脚本应有清晰入口。
- 数据处理过程应可复现。
- 文件路径、采样频率、阈值等参数不应硬编码在算法中。
- 工程计算脚本应保留输入、输出和单位说明。

## 注释规则

注释应解释为什么这样做，或说明关键工程边界。

不推荐：

```text
// 给 x 加 1
```

推荐：

```text
// 通信恢复后等待一个周期再允许输出，避免使用刚恢复时的不稳定数据。
```
